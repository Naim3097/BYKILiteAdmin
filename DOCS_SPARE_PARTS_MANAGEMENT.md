## 1. SYSTEM OVERVIEW

### What It Does
Manages car workshop spare parts inventory. Allows adding, editing, deleting, and searching parts. Tracks stock levels and prices. Parts are also consumed by the Customer Management side (invoices pull parts from this system).

### Collection: `parts`

**Document Schema (exact fields stored in Firestore):**

```
{
  id: string                    // Auto-generated by Firestore (doc ID)
  kodProduk: string             // Product code, e.g. "BRK001"
  namaProduk: string            // Product name, e.g. "Brake Pads - Front"
  harga: number                 // Price in RM (parsed from string via parseFloat)
  supplier: string              // Supplier name
  gambar: string                // Image — stored as base64 DataURL or external URL
  specification: string         // Optional text
  unitStock: number             // Current stock quantity (parsed via parseInt)
  dateAdded: string             // ISO timestamp from new Date().toISOString()
  dateUpdated: string           // ISO timestamp, added on update only
  createdAt: Date               // Added by useFirebaseData hook (new Date())
  updatedAt: Date               // Added by useFirebaseData hook (new Date())
  lastStockChange: {            // Only present after batchUpdateStock()
    reason: string,
    change: number,
    timestamp: Date,
    ...metadata
  }
}
```

**Note**: `createdAt`/`updatedAt` are added by the `useFirebaseCollection` hook automatically. `dateAdded`/`dateUpdated` are added by PartsContext. So documents have BOTH sets of timestamps.

### FUNCTION DETAILS

#### `addPart(part)`

```
Input: { kodProduk, namaProduk, harga, supplier, gambar?, specification?, unitStock }
Process:
  1. Constructs newPart object with parseFloat(harga), parseInt(unitStock), dateAdded
  2. Races: addItem(newPart) vs 1500ms timeout
  3. If Firebase wins → returns { id: firebaseId, ...newPart }
  4. If timeout wins → returns { id: Date.now().toString(), ...newPart }
     (addItem internally saves to localStorage as fallback)
Output: { id, kodProduk, namaProduk, harga, supplier, gambar, specification, unitStock, dateAdded }
```

#### `updatePart(id, updatedPart)`

```
Input: id (string), updatedPart (object with any fields)
Process:
  1. Adds harga: parseFloat, unitStock: parseInt, dateUpdated: ISO timestamp
  2. Calls updateItem(id, updateData)
```

#### `deletePart(id)`

```
Process: Calls deleteItem(id) directly.
```

#### `updateStock(id, quantity)`

```
Process:
  1. Finds part by id in parts array
  2. Calculates new stock: Math.max(0, part.unitStock - quantity)
  3. Updates via updateItem with new unitStock + dateUpdated
```

#### `searchParts(query)`

```
Input: query string
Process:
  - If empty → returns all parts
  - Lowercases query
  - Filters where kodProduk OR namaProduk OR supplier contains query (case-insensitive)
Output: filtered array
```

#### `getLowStockParts(threshold = 10)`

```
Returns: parts.filter(part => part.unitStock <= threshold)
```

#### `batchUpdateStock(stockUpdates)`

```
Input: Array of { partId, newStock, reason?, stockChange?, metadata? }
Process:
  1. Creates Firestore writeBatch
  2. For each update: batch.update(partRef, { unitStock, updatedAt, lastStockChange })
  3. Commits batch atomically
Output: { success: boolean, message: string, updatedParts?: number, error?: string }
```

#### `reserveStock(reservations)`

```
Input: Array of { partId, quantity, sessionId }
Process: Client-side only validation
  - Checks part exists
  - Checks part.unitStock >= quantity
  - Does NOT actually modify stock
Output: { success: boolean, errors: [], reservations: [], message: string }
```

#### `validateStockChanges(changes)`

```
Input: Array of { partId, requiredStock }
Process:
  - If part not found → error (PART_NOT_FOUND)
  - If requiredStock > part.unitStock → error (INSUFFICIENT_STOCK)
  - If remaining stock ≤ 10 → warning (LOW_STOCK_WARNING)
Output: { isValid: boolean, errors: [], warnings: [], canProceed: boolean }
```

**Image Upload System:**

Two methods:
1. **File Upload**: Hidden `<input type="file" id="image-upload" accept="image/*">` with styled label button
2. **URL Input**: Text input for external URL, disabled if file is selected

**File Upload Process (`handleFileSelect`):**
1. Validate: `file.type.startsWith('image/')` → alert if not
2. Validate: `file.size > 5MB` → alert if too big
3. Set `isProcessingImage = true`
4. FileReader reads as DataURL
5. Creates `new Image()` from result
6. On image load (50ms setTimeout to prevent UI blocking):
   - Calculate max dimensions: mobile (<768px) = 400px, desktop = 600px
   - Scale proportionally
   - Draw on canvas
   - Compress: quality = 0.6 (mobile) or 0.8 (desktop), format = JPEG
   - Size check: 400KB (mobile) / 600KB (desktop) limit
   - If too big: retry with quality 0.4
   - If still too big: alert + clearImage()
   - Otherwise: set imagePreview + formData.gambar to DataURL

**URL Input Process:**
- When URL starts with `http`: Creates Image, draws to canvas, converts to JPEG 0.8
- If DataURL > 500KB: alert + clear
- Otherwise: sets imagePreview + formData.gambar

**Validation (`validateForm`):**
- kodProduk: required (trim)
- namaProduk: required (trim)
- harga: required, `parseFloat > 0`
- supplier: required (trim)
- unitStock: required, `parseInt >= 0`
- Sets `errors` state, returns boolean

**Submit (`handleSubmit`):**
1. `e.preventDefault()`
2. Validate → return if false
3. `setIsSubmitting(true)`
4. Call `addPart(formData)` from context
5. Call `onClose()` immediately after (even on timeout errors)
6. Only alerts for real errors (not timeout/offline)
7. `finally`: `setIsSubmitting(false)`

**Modal Structure:**
- Outer: `modal-overlay` class
- Inner: `modal-content sm:max-w-2xl`
- Sticky header with title "Add New Part" + × close button
- Sticky footer with Cancel + Add Part buttons
- Button order reverses on mobile (submit first on mobile)

## 6. HOOKS

### useFirebaseCollection(collectionName) — `src/hooks/useFirebaseData.js`

**This is the core data hook used by ALL contexts.**

**State:**
| Variable | Type | Default |
|----------|------|---------|
| `data` | array | [] |
| `loading` | boolean | true |
| `error` | string\|null | null |
| `retryCount` | number | 0 |
| `isRetrying` | boolean | false |

**Initialization Flow:**
1. Load localStorage cache immediately → if found, set `data` and `loading = false`
2. Set up Firestore real-time listener: `onSnapshot(query(collection, orderBy('createdAt', 'desc')))`
3. Set 1-second timeout — if no Firestore response & no cached data, set `loading = false` with error
4. On snapshot success: update `data`, save to localStorage, clear error
5. On snapshot error: show cached data, set user-friendly error message

**retryConnection():**
- Prevents rapid retries
- Shows cached data immediately
- Exponential backoff: `min(1000 * 2^retryCount, 5000)` ms

**addItem(item):**
1. Adds `createdAt` + `updatedAt` timestamps
2. Generates temp ID: `temp_{timestamp}_{random}`
3. Adds to UI immediately (optimistic)
4. Races: `addDoc()` vs 1-second timeout
5. On success: replaces temp with real ID, updates localStorage
6. On timeout: converts to permanent local item, saves localStorage
7. Returns: Firestore doc ID or local timestamp ID

**updateItem(id, updates):**
1. Adds `updatedAt` timestamp
2. Updates UI immediately
3. Races: `updateDoc()` vs 1-second timeout
4. On success/failure: updates localStorage either way

**deleteItem(id):**
1. Removes from UI immediately
2. Calls `deleteDoc()`
3. Updates localStorage on success or failure

**Returns:**
```
{ data, loading, error, addItem, updateItem, deleteItem, retryConnection, isRetrying }
```

- `createValidationResult(isValid, errors, warnings)` — validation result object
- `createAuditEntry(invoiceId, action, changes, userId)` — audit trail record
- `createContextualError(type, message, context)` — structured error with context

**Net Change Calculation:**
- ADD_PART: `-newQty` (removes from stock)
- REMOVE_PART: `+originalQty` (returns to stock)
- INCREASE_QTY: `-(newQty - originalQty)` (removes difference)
- DECREASE_QTY: `+(originalQty - newQty)` (returns difference)

## 8. PROBLEMS & WHY IT NEEDS TO BE SCRAPPED

Based on the code audit:

### Data Model Issues
1. **No categories/classifications** — parts have no category field (e.g., brakes, engine, electrical)
2. **No brand/manufacturer field** — only "supplier" exists
3. **No minimum stock threshold per part** — hardcoded to 10 across the board
4. **No unit of measurement** — everything assumed to be "units"
5. **No barcode/SKU standard** — `kodProduk` is freeform text
6. **No purchase price vs selling price** — only one `harga` field
7. **No location/bin tracking** — no way to know where parts are stored
8. **No part variants** — can't handle sizes, colors, compatibility
9. **Images stored as base64 in Firestore** — massive documents, slow queries, hits size limits

### Stock Management Issues
10. **No stock history/ledger** — can't trace why stock changed
11. **No purchase orders** — can't track incoming stock
12. **No reorder points per item** — same threshold for a bolt and an engine
13. **Stock reservation is a no-op** — `reserveStock()` validates but doesn't actually reserve
14. **No expiry tracking** — irrelevant for most auto parts but no system for it
15. **No batch/lot numbers** — can't trace defective batches

### Architecture Issues
16. **Duplicate timestamp systems** — `dateAdded`/`dateUpdated` AND `createdAt`/`updatedAt`
17. **localStorage used as primary fallback** — can desync with Firestore, no conflict resolution for parts
18. **1-second timeout on all Firebase ops** — aggressive, causes unnecessary fallbacks
19. **Invoice Number counter in localStorage** — can generate duplicates if multiple tabs/devices
20. **No real-time collaboration** — no locking when two users edit same part
21. **Search is client-side only** — loads ALL parts then filters in memory

### UI/UX Issues
22. **No pagination** — all parts loaded and rendered at once
23. **No sorting** — can't sort by name, price, stock, etc.
24. **No bulk operations** — can't update multiple parts at once
25. **No import/export** — can't bulk upload from CSV/Excel
26. **No print functionality** — can't print stock reports
27. **Delete is permanent** — no soft delete, no undo
28. **No audit trail for parts** — no way to know who changed what
